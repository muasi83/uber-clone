Yes, I can confirm the bug definitively. Here's what's happening:
Root Cause #1: Driver Location Starts As null
When the ride transitions from RiderTrackingScreen → RiderActiveRideScreen, the tracking screen knows the driver's last position (_driverLocation). But it is never passed to the active ride screen.
Arguments passed (from RiderTrackingScreen line 271-278):
rideId, pickupLat, pickupLng, dropoffLat, dropoffLng, dropoffAddress
Missing: driverLat, driverLng, driverHeading, driverId
In RiderActiveRideScreen, _driverLocation stays null until the first WebSocket driver_location event arrives. During that gap — no driver marker is shown at all.
Root Cause #2: No Fallback Fetch for Driver Location
RiderActiveRideScreen.initState() calls _fetchDriverInfo() which fetches driver name + ID from the ride details — but it does not fetch the driver's current GPS location even though the API has it.
Root Cause #3: Car Icon Asset Load Delay (Especially iPhone)
_carIcon = BitmapDescriptor.defaultMarker;  // Initial value (default red pin)
// ...then async:
_carIcon = await BitmapDescriptor.asset(...);  // Takes time on iPhone
On iPhone, loading a BitmapDescriptor from asset files can take 200-800ms. The marker appears with the default icon first (or not at all if location is null), then the car icon "suddenly appears" when the asset finishes loading.
Root Cause #4: Internet Cut-off → No Re-sync
When the rider reconnects after internet loss:
- WebSocket re-establishes a fresh connection
- But the driver's phone doesn't know to re-send its location
- The rider waits for the next periodic driver_location event
- That could be 5-15 seconds of blank screen
The "Circle" You See
It's the rider's own location blue dot (Google Maps default myLocation indicator) — the only marker on screen while the driver marker is absent. The rider sees this and thinks "the driver is approaching as a circle" when it's actually their own position.
How to Fix (Without Changing Code Now)
1. Pass driver location in the arguments from RiderTrackingScreen → RiderActiveRideScreen
2. Add a fallback in _fetchDriverInfo() that also gets the driver's current GPS from the ride details API
3. Pre-load the car icon earlier (in RiderTrackingScreen before navigating) so it's cached
4. Add a driver_location_request WebSocket message — when a rider reconnects, the app sends a request and the server responds with the current driver position
The ngrok factor makes all of these worse (latency), but the root cause is the missing initial data, not the tunnel itself.
